# ADR 0008 — Encryption key strategy

- **Status:** Accepted
- **Date:** 2026-04-11
- **Deciders:** Ersin KOÇ (project lead)

## Context

DeployMonster's secrets module stores tenant-supplied secret values
(database passwords, API tokens, webhook signing keys, cloud-provider
credentials) in the same database as everything else. Those values
must be encrypted at rest so that a stolen `deploymonster.db` file is
not equivalent to a full credential dump. The JWT auth path needs a
signing secret for the same reasons, and the webhook-outbound path
needs HMAC material for signature verification.

At least three separate questions needed answering before writing any
code:

1. **Where does the master key live?** In the config file, an
   environment variable, a separate keyfile, a KMS, or auto-generated
   on first boot?
2. **How is the per-secret data encryption key derived from the master?**
   Direct reuse, HKDF, Argon2id, something else?
3. **How is the ciphertext formatted, and can it be rotated?**

The answers had to satisfy a self-hosted user running on a single VPS
(no external KMS, no hardware HSM, no external secret-manager service)
while still producing ciphertext that is not trivially compromised by
filesystem access alone.

## Decision

### Key storage

The master key is resolved in this order at boot
(`internal/secrets/module.go:60-62`):

1. `secrets.encryption_key` in `monster.yaml` (or `MONSTER_ENCRYPTION_KEY`
   environment variable) — the dedicated vault master.
2. `server.secret_key` in `monster.yaml` (or `MONSTER_SECRET`) — the
   JWT signing key, used as a fallback so single-key setups are
   possible without sacrificing security.

On first boot with no key set, `internal/core/config.go:421-422`
auto-generates a random 32-byte base64 value via `GenerateSecret(32)`
(`internal/core/id.go:20`) and writes it back to the config file. This
keeps the zero-config install story alive without shipping a hardcoded
default.

Operators who want defense in depth can set
`secrets.encryption_key` to a distinct value from `server.secret_key`
so a JWT-key compromise does not immediately decrypt the vault.

### Key derivation

Per-deployment, a 32-byte random salt is generated by
`GenerateVaultSalt()` (`internal/secrets/vault.go:37-43`) on first
boot and persisted base64-encoded in BBolt under `vault/salt`
(`internal/secrets/module.go:19-20`). That salt is passed into
`argon2.IDKey(masterSecret, salt, time=1, memory=64*1024, parallelism=4, keyLen=32)`
to derive the actual AES-256 encryption key
(`internal/secrets/vault.go:67`). Argon2id is used rather than
PBKDF2 or scrypt so the KDF remains memory-hard and GPU-unfriendly,
and rather than HKDF so an attacker with the salt still faces the full
Argon2id cost on every guess.

### Ciphertext format

Secrets are encrypted with AES-256-GCM using a fresh 12-byte nonce
from `crypto/rand` per encryption (`internal/secrets/vault.go:74-90`).
The on-disk ciphertext is `[nonce (12 bytes)] [sealed output]`, the
whole blob base64-encoded. Two different encryptions of the same
plaintext produce different ciphertexts because the nonce is random.

### Legacy salt migration

A single compatibility path exists: pre-Phase-2 installs used a
hardcoded salt string (`"deploymonster-vault-salt-v1"` hashed with
SHA-256 to 16 bytes — see `vault.go:25-32`). On first boot after
upgrade, `migrateLegacyVault()` (`internal/secrets/module.go:176-213`)
detects the absence of `vault/salt`, generates a fresh random salt,
re-encrypts every existing secret version with the new key, and
persists the salt. The migration is idempotent — pinned by
`TestMigrateLegacyVault_IdempotentOnRestart`
(`internal/secrets/vault_salt_test.go:305`) — so a crash mid-migration
does not corrupt data.

### Rotation

Master-key rotation is a supported, single-step operation exposed by
`Module.RotateEncryptionKey(ctx, newMasterSecret)`
(`internal/secrets/module.go:305-350`). It iterates every secret
version, decrypts it with the current vault, re-encrypts it with the
new vault, and writes it back, then atomically swaps the module's
vault instance. Unlike JWT rotation, there is no "previous key"
grace window for the vault: a successful `RotateEncryptionKey` means
all ciphertext is immediately bound to the new master. Tests pin the
full cycle in `internal/secrets/module_coverage_test.go:509-578`.

## Consequences

**Positive:**

- **Zero-ceremony install.** First boot generates a key, writes the
  config, and works. No "before you start, install Vault" detour.
- **Defense in depth is optional.** Operators who care can set a
  distinct `secrets.encryption_key`; the default single-key setup is
  still safer than plaintext-on-disk.
- **Memory-hard KDF.** Argon2id with 64 MiB memory cost means even a
  leaked database file plus a leaked short password requires real
  hardware to brute-force — unlike PBKDF2, which GPUs tear through.
- **Rotation is a first-class operation.** `RotateEncryptionKey` is
  covered by tests and exposed via the CLI (`deploymonster rotate-keys`
  — `cmd/deploymonster/main.go:242-250`). Key compromise has a real
  remediation path, not a hand-wave.
- **Legacy-install migration is non-destructive and idempotent.** The
  pre-Phase-2 hardcoded salt does not lock users out of upgrades.

**Negative / trade-offs:**

- **Single master key at rest.** An attacker with both the database
  file and the config file (or the env) decrypts everything. We
  mitigate with file-permission guidance and recommend environment
  variables over config-in-git — but on a self-hosted VPS there is
  no KMS tier to hide behind.
- **No per-tenant key isolation.** Every tenant's secrets are
  encrypted with the same master. A full-cluster compromise is
  equally bad for all tenants. A per-tenant KDF branch is possible
  but not implemented — noted in "Revisit if".
- **No ciphertext algorithm version marker.** The current format is
  nonce-prefixed AES-256-GCM with no version byte. Swapping to a
  different AEAD in the future requires either a parallel decrypt
  path keyed on a new format marker or a full re-encryption pass
  like `RotateEncryptionKey`. This is acceptable today because we
  have exactly one algorithm, but a future ADR should add a version
  byte before we add a second.
- **No previous-key grace window for vault rotation.** If a
  `RotateEncryptionKey` run is interrupted halfway, some versions are
  on the old key and some are on the new one. The rotation code
  handles this correctly on retry (it decrypts with whichever vault
  works), but the operator-facing story is "don't interrupt it."

## Revisit if

- We ship a hosted offering where KMS integration (AWS KMS, Google
  Cloud KMS, HashiCorp Vault) becomes mandatory for compliance —
  introduce a `core.KeyManagementService` interface that the vault
  can optionally wrap.
- We add a compliance requirement for per-tenant key isolation (e.g.,
  BYOK for enterprise tenants) — add a tenant-scoped salt branch.
- We need a second encryption algorithm (e.g., ChaCha20-Poly1305 for
  an ARM fast path) — add a ciphertext version byte in a new ADR
  before writing any code.
- We add audit-log encryption. The audit log currently stores
  actor/action/target unencrypted, which is fine for self-hosted but
  not for every compliance regime.
